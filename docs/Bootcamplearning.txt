1. Purpose &amp; Context
This document provides a structured, phased learning plan for building enterprise-grade Retrieval-
Augmented Generation (RAG) capabilities within the AI-PMS platform. It is designed for K. Bala
Chowdappa (NLP lead, GPREC) and the students working under his guidance.
The plan transitions from the initial AntiGravity IDE prototype to production-ready RAG pipelines that will
power the Contract Agent (A04), Document Agent (A13), NL Query chatbot, and all other agents requiring
semantic search over AI-PMS’s unstructured content.
1.1 What This Plan Is NOT
• A theoretical NLP course — this is hands-on, experiment-driven, and anchored to real AI-PMS
entities
• A reproduction of the AntiGravity IDE approach — we are moving FROM drag-and-drop TO code-
first, observable, testable pipelines
• An excuse to skip fundamentals — every shortcut in RAG compounds into production failures

4.2 The Breaking Experiments
For each experiment below, the team must write a 1-page finding report: what happened, why it
happened, what it teaches about production RAG.
4.2.1 Experiment: Query-Document Mismatch
Query: “What is the time-bar period for delay claims under FIDIC Yellow Book?”
Expected failure: If chunks are from FIDIC Red Book, the system will hallucinate an answer using Red
Book clauses as if they were Yellow Book clauses. The LLM will NOT say ‘I don’t know.’
Observation: Does the system cite the source contract type? Does it distinguish between
Red/Yellow/Silver Book? This is CRITICAL for the Contract Agent (A04) which must retrieve FIDIC-
specific clauses per contract_standard field.
4.2.2 Experiment: Cross-Entity Confusion

Load NCR descriptions AND contract clauses into the same vector space. Query: “What corrective action
should be taken for concrete honeycombing?”
Expected failure: The retriever will mix NCR corrective actions with contract clauses about defect
liability. The answer will blend operational advice with legal obligations.
Lesson: This is why metadata filtering and namespace separation matter. AI-PMS must tag every chunk
with its entity_type, package_id, and tenant_id.
4.2.3 Experiment: The Long-Document Problem
Load a full 100-page contract. Query: “Summarize the contractor’s obligations.”
Expected failure: Top-K retrieval (K=5) returns 5 chunks from a 100-page document. The ‘summary’ is
based on 2% of the document. This is not summarization; it is sampling bias.
Lesson: Different query intents need different retrieval strategies. Factoid queries work with top-K.
Summarization queries need document-level strategies (map-reduce, refine).

• Retrieval metrics: Precision@5, Precision@10, Recall@10, MRR (Mean Reciprocal Rank),
NDCG@10
• Answer quality: Faithfulness (is the answer grounded in retrieved chunks?), Relevance (does it

�� PITFALL: Optimizing Retrieval Without Measuring Answer Quality
A retriever that returns all relevant chunks but in the wrong order can still produce terrible answers.
The LLM has a recency bias (favors chunks at the end of the context) and a primacy bias (favors
chunks at the start). Measure BOTH retrieval quality AND answer quality. They are not the same
thing.

6.2 Metadata Schema for Retrieval
Every chunk stored in pgvector MUST have the following metadata fields (enforced at ingestion, not
optional):
• tenant_id (MANDATORY — multi-tenancy isolation via RLS)
• entity_type (contract_clause | correspondence | dpr | ncr | meeting_minutes | drawing |
incident_report)
• package_id (which contract package this belongs to)
• source_document_id (traceable back to CDM Document entity)
• chunk_index (position within document for ordering)
• parent_chunk_id (for hierarchical retrieval)
• created_at / updated_at (for temporal filtering — ‘latest revision only’)

7. Phase 4: Agentic RAG &amp; Multi-Hop Retrieval (Weeks 10–11)
This phase connects RAG to the AI-PMS agent framework (LangGraph StateGraph with Supervisor
Agent orchestrating domain agents).
7.1 Agentic RAG Pattern
Instead of a single retrieve-then-generate cycle, an agent decides WHAT to retrieve, evaluates the
results, and iterates:
User Query → Agent plans retrieval steps → Step 1: retrieve contract clauses → Evaluate: enough info?
→ No → Step 2: retrieve correspondence → Evaluate: enough? → Yes → Generate answer with citations
7.1.1 Implementation with LangGraph
Build a LangGraph StateGraph where:
• State includes: query, retrieved_chunks (list), retrieval_history, confidence_score, iteration_count
• Nodes: query_analyzer, retriever, evaluator, answer_generator, citation_builder
• Edges: evaluator → retriever (loop if insufficient) OR evaluator → answer_generator (if sufficient)
• Guard: max 3 retrieval iterations to prevent infinite loops
7.2 Multi-Hop Queries — The Real Test
These are the queries that justify the agentic approach:
Multi-Hop Query Example Required Retrieval Steps
“Is the contractor at risk of a time-bar miss
on Package CC-07?”

Step 1: Retrieve notices for CC-07. Step 2: Retrieve
FIDIC time-bar clauses. Step 3: Compare notice dates

“What NCRs are related to activities on

Step 1: Retrieve latest DPRs per package (temporal +

(structured data, not RAG). Step 3: Generate comparison

✅ KEY INSIGHT: RAG is Not Always the Answer
Some AI-PMS queries need SQL (quantitative: ‘What is the CPI for CC-07?’), some need graph
traversal (relational: ‘What depends on this activity?’), some need vector search (semantic: ‘Find
similar NCRs’), and many need a combination. The agent’s query analyzer must ROUTE to the right
retrieval strategy, not default to vector search for everything.
