1. The No-GPU Strategy
The GPREC HPC cluster does not currently have GPUs provisioned for the RAG bootcamp. This is not a
blocker. The bootcamp can run entirely on free cloud APIs for LLM inference and on CPU for embeddings.

3. What Runs Locally on CPU (No API Needed)
3.1 Embedding Models on CPU
Sentence-transformers embedding models are small enough to run on CPU. Slower than GPU but perfectly
usable for bootcamp-scale data (1000–5000 chunks).
pip install sentence-transformers
from sentence_transformers import SentenceTransformer
# These models run fine on CPU:
model = SentenceTransformer(&quot;all-MiniLM-L6-v2&quot;) # 80MB, 384-dim, ~100 chunks/sec on
CPU
# model = SentenceTransformer(&quot;BAAI/bge-large-en-v1.5&quot;) # 1.3GB, 1024-dim, ~20 chunks/sec
on CPU
# model = SentenceTransformer(&quot;nomic-ai/nomic-embed-text-v1.5&quot;) # 550MB, 768-dim
chunks = [&quot;tunnel boring machine specifications&quot;, &quot;OHE catenary installation&quot;, ...]
embeddings = model.encode(chunks, show_progress_bar=True, batch_size=32)
print(f&quot;Generated {len(embeddings)} embeddings of dimension {embeddings.shape[1]}&quot;)

3.2 Cross-Encoder Reranking on CPU
Cross-encoders are more compute-intensive than bi-encoders but still viable on CPU for small batch sizes (top-
30 candidates).
from sentence_transformers import CrossEncoder
reranker = CrossEncoder(&quot;cross-encoder/ms-marco-MiniLM-L-12-v2&quot;) # ~130MB
# Rerank top-30 retrieved chunks
query = &quot;What is the time-bar period for FIDIC delay claims?&quot;
pairs = [(query, chunk) for chunk in retrieved_chunks[:30]]
scores = reranker.predict(pairs) # ~2-5 seconds on CPU for 30 pairs
# Sort by reranking score
ranked = sorted(zip(scores, retrieved_chunks[:30]), reverse=True)
top_5 = [chunk for _, chunk in ranked[:5]]

⚠️ CPU LATENCY NOTE
Reranking 30 chunks takes ~2–5 seconds on CPU vs. ~200–500ms on GPU. For the bootcamp this
is fine. For production, GPU reranking will be needed. Record the CPU latency in your experiment
logs so the architecture decision document captures this trade-off.
3.3 Everything Else (CPU, No API)
These components are all CPU-native and need zero cloud APIs:
• PostgreSQL + pgvector — vector similarity search runs on CPU. HNSW index for speed.
• PostgreSQL pg_trgm + tsvector — BM25 keyword search. Pure CPU.
• Apache AGE — graph traversal on CPU. No GPU needed for graph queries.
• ChromaDB — in-memory vector store for rapid prototyping. CPU only.
• FAISS (faiss-cpu) — Facebook’s ANN library, CPU version. For benchmarking.
• LangChain / LangGraph / LlamaIndex — orchestration frameworks. CPU.
• RAGAS — evaluation framework. Uses LLM API calls (Groq) for evaluation, not local GPU.
• MLflow — experiment tracking. Web server on CPU.

4. Building a Fallback Chain (Important!)
Free APIs have rate limits. Build a fallback chain so your experiments don’t crash when one provider hits its
limit.
from openai import OpenAI
import time
# Configure all providers
PROVIDERS = {
&quot;groq&quot;: {
&quot;base_url&quot;: &quot;https://api.groq.com/openai/v1&quot;,
&quot;api_key&quot;: os.environ.get(&quot;GROQ_API_KEY&quot;),
&quot;model&quot;: &quot;llama-3.3-70b-versatile&quot;
},
&quot;openrouter&quot;: {
&quot;base_url&quot;: &quot;https://openrouter.ai/api/v1&quot;,
&quot;api_key&quot;: os.environ.get(&quot;OPENROUTER_API_KEY&quot;),
&quot;model&quot;: &quot;deepseek/deepseek-chat-v3-0324:free&quot;
},
&quot;cerebras&quot;: {
&quot;base_url&quot;: &quot;https://api.cerebras.ai/v1&quot;,
&quot;api_key&quot;: os.environ.get(&quot;CEREBRAS_API_KEY&quot;),
&quot;model&quot;: &quot;llama-3.3-70b&quot;
}
}
def query_llm(messages, provider_order=[&quot;groq&quot;, &quot;openrouter&quot;, &quot;cerebras&quot;]):
&quot;&quot;&quot;Try each provider in order. Fall back on 429 or error.&quot;&quot;&quot;
for provider_name in provider_order:
config = PROVIDERS[provider_name]
try:
client = OpenAI(base_url=config[&quot;base_url&quot;], api_key=config[&quot;api_key&quot;])
response = client.chat.completions.create(
model=config[&quot;model&quot;],
messages=messages,
temperature=0.1,
max_tokens=1024
)
return response.choices[0].message.content, provider_name
except Exception as e:
print(f&quot;{provider_name} failed: {e}. Trying next...&quot;)
time.sleep(1)
raise Exception(&quot;All providers failed&quot;)
# Usage
answer, provider = query_llm([{&quot;role&quot;: &quot;user&quot;, &quot;content&quot;: &quot;Your RAG query here&quot;}])
print(f&quot;Answer from {provider}: {answer}&quot;)

✅ THE SILVER LINING: You Get a Better LLM for Free
Groq gives free access to Llama 3.3 70B — which is significantly more capable than the Llama 3.1
8B you would have run locally. Your RAG answers and RAGAS evaluations will actually be BETTER
quality during the bootcamp than they would have been with a local 8B model. The trade-off is
latency (network round-trip) and rate limits.
6.1 Latency Caveat
Because LLM inference goes over the network to Groq/Google, the latency numbers in your experiments will
NOT reflect production performance on GPREC hardware. In every experiment log, note:
• Retrieval latency (local pgvector) — THIS IS MEANINGFUL, record it carefully
• Reranking latency (local CPU) — MEANINGFUL but will be faster on GPU, note the CPU penalty
• LLM generation latency (Groq API) — NOT MEANINGFUL for production. Record it but flag it as ‘cloud
API, not production representative’
• End-to-end latency — Report but note that LLM component will change in production
