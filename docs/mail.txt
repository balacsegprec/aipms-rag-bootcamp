DATASETS — THREE CORPORA TO USE:

Since FIDIC books are copyrighted and not freely available, the team should use three data sources:

a) Kaggle Enterprise RAG Dataset (for general RAG skills)
URL: https://www.kaggle.com/datasets/rrr3try/enterprise-rag-markdown
This contains 100 real annual reports from the Enterprise RAG Challenge, already converted to markdown. Complex documents with tables, numbers, multi-page content. Use this for the early phases: embedding comparison, chunking experiments, naive RAG pipeline, breaking experiments. It will give the team real-world messy documents to learn on.

b) Indian Railways GCC (for contract clause experiments)
The Indian Railways Standard General Conditions of Contract (GCC 2022) is a government document, freely available. It has the same clause-subclauses hierarchical structure that our contract chunker needs to handle. Download from:
• IRICEN: https://www.iricen.gov.in/iricen/Works_Manuals/GCC_July_2020.pdf
• DFCCIL GCC: https://dfccil.com/upload/2_GCC_OOOA.pdf
• IREPS GCC 2022: https://www.ireps.gov.in/ireps/upload/repository/railway/3482/456083/private/GCC-2022.pdf

Use these for the contract clause chunking experiments. They test the same hierarchical parsing challenges as FIDIC (clause 20.1, sub-clause 20.1.1) and are directly relevant to our IR/DFCC tenant.

c) Synthetic DMRC Dataset (for AI-PMS domain-specific work)
Our existing AI-generated DMRC Mega Metro data. Use this for domain-specific phases: document-type-specific chunkers (NCRs, DPRs, correspondence, meeting minutes), metadata filtering, tenant isolation testing, and GraphRAG on the Systems Taxonomy. Remember — valid for pipeline testing only, not for model training or threshold calibration.


KEY POINTS FOR CHOWDAPPA SIR:

• The plan starts with fundamentals before any framework usage. The initial phase is plain Python only — no LangChain, no LlamaIndex. The team must understand what frameworks abstract away before using them.

• The breaking experiments are the most important part. The team must intentionally break the naive pipeline and document WHY it broke. Skipping this step means building pipelines they cannot debug.

• Every experiment must be logged in the structured format (Section D10 of the Deliverables doc). No experiment without metrics. No metric without an observation.

• The bootcamp plan includes a suggested student assignment matrix with parallel tracks (retrieval strategies, evaluation pipeline, document parsers, infrastructure, query datasets). Once students are assigned, Chowdappa Sir can distribute work based on their skill levels using this as a guide.

• Since there is no GPU, all LLM calls go through Groq’s free API (Llama 3.3 70B). The No-GPU Guide has complete setup instructions and code. The silver lining: Groq gives free access to a 70B model, which is much better than the 8B model we would have run locally.

• One caveat: LLM latency measurements over cloud APIs are NOT production-representative. The team should record them but flag as ‘cloud API’ in every log. Retrieval and reranking latency on local CPU IS meaningful.
